MARK IV

SCHED  supports Mark IV observations on Mark IV (and VLBA4) antennas equipped with the Field System (PCFS). This is done by creating a *.skd file in the VEX format. The VEX file is a global and self-contained description of the experiment that is scheduled. The *.skd file can be read by the control system (PCFS) and translated (by a program called DRUDG) into a list of commands for the telescope, data acquisition system and recorder.

SCHED  produces VEX files with ``adaptive'' tape control that enables continuous tape motion on PCFS controled telescopes, but can only be correctly interpreted by FS9.3.11 (and higher).

In order to cause SCHED  to write VEX files, the user must either include in the schedule one telescope for which CONTROL = VEX is specified in the station catalog or must specify the input parameter DOVEX. Two example schedules that produce VEX files are included with the SCHED  distribution and are located in the $SCHED/examples subdirectory. They are called egiv.key and egvsop.key.

Mark IV (MkIV) telescopes are characterized by DAR = MKIV and RECORDER = MKIV in the station catalog and can only be requested to record in any of the MkIV FORMATs. Most MkIV telescopes only support specific BBC's to be connected to certain IF distribution boxes. Currently SCHED  only supports ``astronomical patching''. This requires odd BBC's to be connected to either IFCHAN = 1N (normal input, usually LCP) or IFCHAN = 1A (alternate, usually RCP). Even BBC's have to be connected to IFCHAN = 2N (usually RCP) or IFCHAN = 2A for LCP. This can be controlled by explicitly setting BBC.

In December 2000 the capability to record 512 Mb/s was introduced. This is achieved by recording 64 tracks simultaneously. It requires NHEADS = 2 in the station catalogue. Many MkIV stations now record using the Mk5 disk-based system. MkIV Stations equipped with Mk5 systems can record data at up to 1 Gbps. This requires DISK = MARK5A in the station catalogue.

Several features are not yet available or supported for MkIV (and MarkIV based Mark5 systems). Some as of late 2004 are:

  • Continuous tape motion has been used widely, however the ability to acquire the information needed to flag data which is recorded during slewing is not universal. The SCHED  output .flg file can be used by the AIPS task UVFLG. In addition users need to insert gaps during continuous tape motion whenever they want calibration data. Finally, at least 40 seconds are required to turn the tape around.

  • Phase Cal detection at the telescopes is not implemented at all telescopes. This requires soft- and hardware upgrades at the telescopes, the field system, and probably SCHED.

  • For MkIV (VLBA4) telescopes there is flexibility for controling when Tsys measurements will be made. Tsys measurements will typically be made at the beginning of a scan if there is a tape stop. When scheduling long scans or continuous tape motions, one may need to insert gaps in order to obtain Tsys data.

SCHED has the ability to discourage recording modes that are not supported by one or the other of the VLBA or Mark IV (VLBA4) systems. Currently (Oct. 2001) the supported modes on both systems are the same so this facility has no effect, except for some Mark IV (and VLBA4) features that are experimental. An override (MODETEST) is available but should be used with caution.

In addition to requiring use of tested modes, SCHED  will not allow speedup factor changes during mixed VLBA and Mark IV mode observations. This is because of tape speed dependent delays that are different in the two formats. The correlator would require different clocks for different modes, which is not yet available. Fairly soon, this restriction will be relaxed. Again, MODETEST will turn off the test.

Eventually it will be required that NRAO have some way to read Mark IV VEX format *.skd files and produce VLBA control files. This might be done using VEX input to SCHED  or by supporting general types of observations with DRUDG, which already serves this function for geodesy projects. Probably nothing will be done in this area until some program other than SCHED starts being used for general VLBA scheduling.
